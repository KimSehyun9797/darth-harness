# 안전한 에이전트 시스템이 너무 느려졌을 때

Darth Harness(개발명 agent-harness) 오케스트레이션 효율 개선 사례

## 한 문장 요약

여러 AI 에이전트를 안전하게 지휘하는 체계는 만들었지만, 모든 작은 작업에 같은 큰
절차를 반복하면서 시간과 토큰을 과도하게 썼다. 검증을 없애는 대신 작업 등급,
선택적 문맥, 검증 영수증, 하드 예산으로 반복 비용이 생기는 위치를 바꿨다.

## 시작한 이유

긴 코딩 작업을 AI 하나에 맡기면 네 문제가 반복됐다.

1. 세션이 끝나면 왜 그런 결정을 했는지 사라졌다.
2. “완료”라는 말과 실제로 검사된 결과를 구분하기 어려웠다.
3. 여러 에이전트를 쓰면 누가 무엇을 하는지 보이지 않았다.
4. 자동화가 강해질수록 사람의 범위·안전·공개 결정권이 약해졌다.

목표는 단순했다. 모델은 바뀌어도 일하는 방식은 유지하고, 결과는 말이 아니라
증거로 판정하며, 사용자가 모든 워커를 볼 수 있게 한다.

## 처음 만든 구조

하네스를 세 층으로 나눴다.

- 환경 층: 독트린, 모델 역할표, 실행·검사 스크립트, 프로젝트 template
- 프로젝트 층: 목표, 작업 의존성, 워커 브리프, 상태, 실행·검증 증거
- 지식 층: 프로젝트가 끝난 뒤 재사용할 배움

사람은 목표와 제약을 정한다. 오케스트레이터는 작업을 나눈다. 워커는 제한된 파일을
수정한다. 저지는 결과와 근거를 따로 심사한다. Git과 상태 파일이 이 과정을 보존한다.

## 첫 왕복이 보여준 결함

초기 T1~T10 구현 뒤 Claude와 Codex 워커, 결정론 검사, 독립 저지, 결과 수집,
사용자 GO/KILL까지 실제 왕복했다. 동작은 했지만 바로 두 결함이 드러났다.

### 검증한 코드와 합칠 코드가 같다는 보장이 약했다

워커가 검사한 결과가 있더라도, 그 뒤 파일이 바뀌면 같은 결과라고 볼 수 없다.
B-001은 실행 ID, 종료 코드, 완료 파일, 원시 로그, Git tree와 hash를 한 증거 계약으로
연결했다. 독립 리뷰에서 결함이 발견됐고 수정·재리뷰 뒤 통과했다.

### 사용자 결정 직후 세션이 끝나면 상태가 빠졌다

사용자가 GO를 답한 직후 한도가 끝난 실제 사건이 있었다. B-002는 질문을 보내기
전에 요청 상태를 커밋하고, 다음 입력을 answer와 pending 복구본으로 보존한 뒤에만
모델에 전달하게 했다. 재개는 대화 기억 대신 STATUS, HANDOFF, tasks와 Git을 읽는다.

## 보이는 실행과 상태판

B-004는 워커를 현재 cmux 화면의 pane에 띄웠다. 첫 워커는 오른쪽, 추가 워커는
아래로 배치한다. 상태 확인은 지정한 화면과 worktree Git 상태만 읽고 프로세스 목록을
추측하지 않는다.

Status TUI는 실행기에서 분리한 읽기 전용 프로그램으로 만들었다. 프로젝트와 Queue,
현재 세부 작업, 컨텍스트, 주간 한도, 크레딧, 오케스트레이터와 모든 워커의 실제
모델을 표시한다. 알 수 없는 값은 `?`다.

실제 cmux 80열 화면과 55열×21행 작은 pane을 검사했다. 짧은 pane에서 하단 정보가
사라지는 결함과 설치 링크로 실행할 때 라이브러리를 찾지 못하는 결함도 이 과정에서
발견해 회귀 검사로 고정했다.

## 어디서 비용이 폭증했나

문제는 Status TUI 하나가 아니었다.

> B-002에서 첫 불편 → B-001에서 반복 → B-003에서 심각화 → TUI에서 구조 문제 확정

보존 기록에서 확인한 수치는 다음과 같다.

| 구간 | 관찰값 |
|---|---:|
| B-002 설계 | 352줄 |
| B-002 구현 계획 | 1,651줄 |
| B-002 설계부터 main 통합 | 약 10시간 38분 |
| B-003 Task 1~3 워커 입력·출력 토큰 합계 | 762,915 |
| B-003 워커 로그 | 약 72,000줄, 약 4MB |
| B-003 전체 테스트 기록 | 최소 23회 |
| 초기 TUI 실행 스캐폴드 | 1,243줄 |
| TUI 설계·계획 | 1,266줄 |
| TUI producer P1 제품 코드·테스트 | 320줄 |
| TUI 제어 체크포인트 한 번 | 로그·증거 약 2,902줄 추가 |

숫자의 핵심은 “AI 작업은 원래 비싸다”가 아니다. 제품 코드보다 계획, 반복 검사,
원시 로그, 상태 복제가 더 커졌다는 점이다.

## 근본 원인

1. 프로젝트가 복잡하다는 이유로 거의 모든 하위 작업을 T2로 분류했다.
2. 각 노드가 계획, 브리프, 전체 검사, 완료 파일, 중앙 검증, 리뷰를 반복했다.
3. 코드와 명령이 같아도 검증 결과를 재사용하지 않았다.
4. 긴 로그와 과거 문서를 다음 모델 입력에 다시 넣었다.
5. 상태 사실을 여러 문서에 손으로 복제했다.
6. 일반 구현에도 고비용 모델을 라우팅했다.
7. 권한·worktree·공급자 실패 뒤 좁게 재개하지 않고 처음부터 반복했다.
8. 변경량과 모델 호출이 커져도 실행기를 멈추는 하드 게이트가 없었다.

인과 관계는 다음과 같았다.

```text
복잡한 프로젝트
  → 모든 하위 작업을 복잡하게 취급
  → 노드마다 전체 절차 반복
  → 전체 출력과 문서 재주입
  → 시간, 토큰, Git 기록 폭증
```

## 하지 않기로 한 것

### 모든 작업에 30분 제한

작업 크기와 위험이 다른데 같은 시간을 강제하면 필요한 안전 검사가 잘리거나 예외가
숨는다. 시간 하나가 아니라 작업별 실행, 검사, 토큰, 변경량 상한이 필요했다.

### 테스트와 리뷰 축소

반복 비용은 줄지만 하네스의 존재 이유도 사라진다. 최종 전체 회귀와 T2/T3 독립
리뷰는 유지했다.

### 과거 문서와 로그 전체 삭제

재개와 감사 근거를 잃는다. 삭제 대신 매번 읽는 hot 문맥과 필요할 때만 읽는 cold
문맥으로 분리했다.

### Ponytail 전체 설치

필요성부터 따지는 Lean Gate는 유용했다. 하지만 전체 훅과 규칙을 다시 주입하면
우리 작업 계약과 이중 관리된다. 판단 순서만 실행 계약에 직접 넣었다.

## 외부 사례에서 가져온 것

### multi-agent-starter

- 모델이 필요 없는 생성·검사를 결정론 코드로 처리
- 안정적인 역할과 바뀌는 실제 모델을 분리
- 가짜 백엔드로 유료 모델 없는 빠른 회귀

### Knot

- 규칙 정본은 한 곳에 두고 연결 파일은 얇게 유지
- 색인부터 읽고 필요한 자료만 늦게 로드
- 원본과 에이전트 해석물을 분리

### Ponytail

- 필요성 → 기존 코드 → 표준 기능 → 설치된 도구 → 최소 구현 순서
- 안전, 보안, 데이터 손실 방지는 절감 대상에서 제외

셋 중 하나를 그대로 복사하지 않았다. 기존 하네스의 안전 원칙 아래 필요한 부분만
결합했다.

## 실제로 바꾼 구조

### 1. 작업 등급과 모델 역할

T0~T3로 위험과 절차를 나눴다. 기본 워커는 1명, 동시에 최대 3명이다. 일반 구현은
일반 워커, 단순 작업은 저비용 워커, 복잡한 설계와 저지는 상위 역할에 맡긴다.

실제 공급자와 모델은 `MODELS.yaml` 한 곳에서 바꾼다. 작업 계약은 모델 이름이 아니라
역할을 참조한다.

### 2. 계약 내장 Lean Gate

실행 동작을 바꾸는 작업은 구현 전에 가장 작은 해법과 근거 한 문장을 기록한다.
누락, 잘못된 자료형, `not-needed`와 상태 불일치는 dispatch 전에 거부한다.

### 3. 검증 영수증과 안전한 재사용

결정론 검사만 코드, 명령, 선언 환경의 지문이 같을 때 재사용한다. 원시 로그는 로컬
cold 영역에 두고 Git에는 짧은 영수증을 남긴다. 시간과 네트워크에 의존하는 검사는
재사용하지 않는다.

### 4. 선택적 문맥

워커는 브리프와 최대 5개 hot 파일부터 읽는다. 합계 200줄·32KiB를 넘기지 않는다.
전체 스펙, 과거 설계, 원시 로그와 블로그 원자료는 cold다. 현재 정보로 완료 기준을
판단할 수 없을 때만 연다.

### 5. 실제 하드 예산

모델 실행, 수정 반복, 관련·전체 검사, 토큰, 변경 파일과 줄 수에 상한을 둔다.
초과하면 `BUDGET_STOP`이 dispatch와 검사를 모두 막는다. 중단 파일과 상태를 먼저
checkpoint하고 새 계약이 있을 때만 재개한다.

### 6. 읽기 쉬운 Status TUI

프로젝트 목적, Queue 1–9, 현재 하위 작업, allowance와 agent를 분리해 표시한다.
Codex 크레딧 잔액과 변화량도 공급자가 확인한 값이 있을 때 보여준다. UserPromptSubmit
훅이 매 하네스 세션에서 상태 pane을 시작하거나 재사용한다.

### 7. private 전용 첫 백업

새 프로젝트는 저장소를 만들기 전에 올라갈 전체 파일과 확인 지문을 출력한다. 같은
지문일 때만 private 저장소를 만들고 push한다. 실패 후 원격을 삭제하거나 force push로
우회하지 않고 같은 저장소에서 재시도한다.

## 무엇이 검증됐나

| 대상 | 최신 로컬 증거 | 무엇을 증명하는가 |
|---|---:|---|
| darth-harness | PASS=354, FAIL=0 | 실행 계약, 상태, 예산, 이관, TUI producer, private 백업 회귀 |
| cmux-harness-status | PASS=29, FAIL=0 | 파싱, 작은 화면, 모델·한도 표시, 설치 링크 실행 |
| 실제 cmux QA | 과거 80열과 55×21 통과 | 최신 자동 시작 재검사에서는 pane 유지 실패, 회귀 조사 중 |
| private 백업 집중 검사 | PASS=8, FAIL=0 | 공개 거부, 파일 지문, 생성·push, 실패 후 재시도 |

실제 GitHub CLI로 현재 PUBLIC 저장소를 읽었을 때 private 도구가 push 전에 거부하는
것도 확인했다. 원격에는 아무것도 쓰지 않았다.

## 아직 증명되지 않은 것

- 같은 작업을 개선 전후로 반복한 시간·토큰 절감률
- Linux와 Windows의 동일한 운영 경험
- tmux에서 cmux와 같은 한 화면 pane 배치
- 공급자 상태 데이터가 없을 때도 완전한 사용량 표시
- B-003 선제 공급자 교대의 남은 단계
- 내부 working history와 public release의 최종 배포 방식

그래서 이 문서는 “더 싸고 빠르다”고 보장하지 않는다. 반복 비용의 원인을 코드 구조로
제거했고 기존 안전 검사가 유지된다는 것까지 증명한다.

## 최종 성숙도 판단

새 프로젝트를 만들고, 목표를 계약으로 바꾸고, 보이는 워커에게 맡기고, 결과를
검증하고, 세션을 넘겨 재개하는 핵심 루프는 작동한다. Status TUI와 private 첫 백업도
실행 경로에 연결됐다.

아직 완전 자율 시스템은 아니다. 공개, 예산 확대, 범위 변경, 위험 감수, 최종 GO/KILL은
사람이 결정한다. 이 제한은 미완성의 흔적이 아니라 제품의 안전 경계다.

## 다음 검증 우선순위

### 반드시

1. private working history와 clean public release의 분리 방식을 확정한다.
2. 공개 대상에서 로컬 경로, 운영 HANDOFF, 사용량 영수증, 커밋 이메일을 정리한다.
3. 공개 후 새 컴퓨터에서 clone → install → start 왕복을 다시 실행한다.

### 권장

1. 같은 유형의 T1·T2 작업을 개선 전후 조건으로 반복한다.
2. 시간, 입력·출력 토큰, 검사 실행 수, Git 기록량을 함께 측정한다.
3. 실패율과 독립 리뷰 결과가 나빠지지 않았는지 비교한다.

### 나중

1. 보류된 공급자 선제 교대의 가치를 다시 평가한다.
2. Knowledge Base와 macOS 정비는 별도 프로젝트로 계획한다.
